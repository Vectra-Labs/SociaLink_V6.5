1. Introduction

1.1 Contexte du Projet

SociaLink est une plateforme web permettant de connecter des travailleurs sociaux indépendants (éducateurs spécialisés, assistants sociaux, psychologues) avec des établissements (EHPAD, crèches, centres d'accueil) au Maroc. Le projet répond à un besoin réel : faciliter le recrutement flexible dans le secteur social.

1.2 Objectifs Pédagogiques

Ce projet de fin de formation démontre ma capacité à :

  • Concevoir et développer une application full-stack complète
  • Implémenter une architecture REST avec authentification sécurisée
  • Gérer des bases de données relationnelles complexes
  • Créer des interfaces utilisateur modernes et responsives
  • Intégrer des services tiers (paiements, emails, temps réel)
  • Appliquer les bonnes pratiques de sécurité web

1.3 Public Cible

  • Workers: Professionnels cherchant missions freelance/CDI
  • Establishments: Structures recrutant des profils sociaux
  • Admins: Équipe validant profils et documents

2. Présentation de SociaLink

2.1 Concept

SociaLink fonctionne comme un LinkedIn spécialisé pour le secteur social marocain. Les travailleurs créent leurs profils, les établissements publient des missions, et le matching se fait via un système de candidatures.

4.2 Authentification JWT

Flux d'authentification:

  1. User envoie email + password → POST /api/auth/login
  2. Backend vérifie credentials (bcrypt compare)
  3. Si valide → génère JWT token (expiration 7 jours)
  4. Token envoyé en réponse + stocké cookie httpOnly
  5. Frontend utilise token dans header Authorization: Bearer {token}
  6. Middleware protect vérifie token sur routes protégées
// Exemple middleware protection routes
export const protect = async (req, res, next) => {
  let token = req.headers.authorization?.split(" ")[1] || req.cookies?.jwt;
  
  if (!token) return res.status(401).json({ message: "Non authentifié" });

  const decoded = jwt.verify(token, process.env.JWT_SECRET);
  const user = await prisma.user.findUnique({ where: { user_id: decoded.id } });
  
  req.user = user;  // User disponible dans contrôleurs
  next();
};

4.3 Contrôle d'Accès par Rôle (RBAC)

Middleware requireRole restreint accès selon rôle utilisateur:

// Exemple protection route admin
router.get('/admin/users', 
  protect,                           // 1) Vérifier authentification
  requireRole(['ADMIN', 'SUPER_ADMIN']), // 2) Vérifier rôle
  getAllUsers                        // 3) Exécuter logique
);
Route Méthode Rôles Autorisés
/api/missions GET Tous (public)
/api/missions/create POST ESTABLISHMENT
/api/applications/create POST WORKER
/api/admin/validate-profile PUT ADMIN, SUPER_ADMIN
/api/super-admin/create-admin POST SUPER_ADMIN uniquement

4.4 Paiements Stripe

Processus abonnement:

  1. Worker choisit plan (BASIC ou PRO) sur frontend
  2. Frontend appelle POST /api/stripe/create-checkout-session
  3. Backend crée session Stripe → URL paiement retournée
  4. Worker redirigé vers Stripe Checkout (paiement sécurisé)
  5. Stripe envoie webhook checkout.session.completed à notre backend
  6. Backend active abonnement en DB + met statut user à VALIDATED

Sécurité Webhooks Stripe

Critical: Les webhooks Stripe sont signés via HMAC SHA-256. Notre code vérifie systématiquement la signature avant traiter événement, empêchant attaquants d'envoyer faux événements (ex: activer PRO gratuitement).

7. Sécurité & Conformité

7.1 Mesures de Sécurité Implémentées

Mesure Implémentation Protection
Hashing Passwords Bcrypt (cost factor 10) Passwords jamais stockés en clair
JWT Tokens Signature HMAC SHA-256 Tokens non falsifiables
HTTPS Production only Chiffrement transport
CORS Origin whitelist Requêtes cross-origin contrôlées
Helmet.js Security headers XSS, clickjacking, MIME sniffing
SQL Injection Prisma ORM (queries paramétrées) Aucune query raw SQL
Email Verification OTP 6 chiffres (15min expiration) Propriété email prouvée
RBAC Middleware requireRole Accès routes restreint par rôle

7.2 Points d'Amélioration Identifiés

Recommandations Sécurité

  • Rate Limiting: Ajouter express-rate-limit (100 req/15min/IP)
  • MFA/2FA: Authentification à deux facteurs (TOTP)
  • Account Lockout: Bloquer compte après 5 tentatives login échouées
  • Logging Centralisé: Winston + CloudWatch pour audit trail
